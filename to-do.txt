## Sentence tagging throughput

- Current code diagnosis:
  - sentence_mythic_sceptic_analyser.py defaults to non-mini gpt-5.
  - Each request sends both the Greek sentence and the English sentence.
  - It makes one API call per sentence, runs sequentially, commits per sentence,
    and sleeps for 0.5 seconds after every call.
  - cronscript.sh currently asks for only 25 sentence classifications per run.
  - The sentence script does not persist token usage, so historical cost/rate
    cannot be audited from the database.
- Live raksasa snapshot on 2026-05-08:
  - 9,715 total sentence rows.
  - 5,442 tagged sentence rows.
  - 4,273 untagged sentence rows.
  - 2,753 untagged sentence rows if books 4 and 8 are excluded.
  - All 9,715 sentence rows have English sentence text.
- Switch the sentence classifier to gpt-5.4-mini for this backlog unless a
  validation sample shows unacceptable drift.
- Add persistent sentence-classification run metadata: model, prompt version,
  input tokens, output tokens, timestamp, and failure status.
- Add a daily budget mode set for 250,000 tokens/day and stop because the budget
  is reached, not because a hard-coded batch of 25 rows is reached.
- Raise the daily batch size substantially after the model switch. If the
  average request stays small, the remaining non-4/non-8 backlog should be a
  few thousand calls, not a long-running trickle.
- Consider modest concurrency or async dispatch once token accounting and retry
  handling are in place; the current 0.5 second sleep is not the main limiter
  if the cron is capped at 25 rows.
- Add a dry-run or sample validation mode so a small gpt-5.4-mini sample can be
  compared against existing gpt-5 labels before retagging everything.

- Make visible the question: which Greek words cause English translations to be
  longer or shorter than expected from Greek length?
- Also add the dual view: which English words are disproportionately found in
  longer-than-expected or shorter-than-expected passages?
- Current code already has a Greek-vocabulary residual model and website page
  generator in website/data.py and website/generators.py, with tests in
  tests/test_translation_length_analysis.py.
- Current visibility gap: pausanias_site/translation_length/index.html is not
  present in this checkout, and the live
  https://pausanias.symmachus.org/translation_length/index.html currently
  returns 404. Rebuild and redeploy the site after confirming the page output.
- Extend the residual model beyond word counts if useful:
  - compare Greek and English word counts first;
  - then consider syllables, letter length, phoneme length, word rarity, or
    Scrabble-like complexity as alternative metrics.
- Use repeated AI translations, where practical, to reduce the objection that a
  residual is just one translator's style or obsession with a particular term.
- Treat Pausanias as the better first corpus than Stephanus for this experiment,
  because Stephanus is already so compressed that compression may dominate the
  result.
- Keep the geography/landscape hypothesis explicit: the likely interesting
  signal is about how ancient and modern language handle landscape, springs,
  cities, built features, and experiential geography.
- Compare against Strabo or Herodotus as possible next corpora once the
  Pausanias pipeline produces useful residual outputs.
